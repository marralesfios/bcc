The pitch

BCC is a structurally-edited language designed to allow developers to write code and collaborate on projects more effectively. The main feature is that source code is stored as a binary format rather than plain text, and all variable/function/type references are by ID instead of string name. While this does pose challenges around integration with existing tooling (diffing, text searching, embedding snippets in websites, various LoC counters, etc.), necessitating the development of alternative, specialized tools, we believe that these costs are outweighed by...

The benefits (click to collapse)

  • No complex name resolution rules

    Name resolution rules in existing programming languages are often convoluted, which not only poses problems to automatic code-generation tools, but also sometimes require the use of hacks to get around name shadowing problems, such as the (in)famous const that = this; in JavaScript.

  • Painless renaming

    Function and type names in stable public APIs are seldom changeable without great cost, and ones later found to be incorrectly designed cannot be directly replaced with new implementations under the same name without breaking old users (especially if the new interface happens to be the same or compatible with the old, where the meaning of legacy code might be silently changed without warning). Victims of this include std::function in the C++ standard library (obsoleted in favor of std::copyable_function and std::move_only_function), Date in JavaScript (to be replaced by various new Temporal classes such as PlainDateTime), and input in Python 2 (extremely unsafe, with raw_input being the safe version)[1]. With ID-based references, we can simply rename the old function, and provide the new interface using the intuitive and obvious name (but a different ID). That way, old code will automatically refer to the legacy interface and still keep working.

  • More efficient compilation

    When the source code is already in AST form, the compiler doesn't need to scan over it again to parse it into a machine-readable format. Additionally, the editor tightly integrates with the compiler backend, and type inference calculations (for example) performed for code highlighting and autocomplete are directly reused in the backend, which reduces the amount of redundant work done.
